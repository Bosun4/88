## 5. 重点样本：塞尔塔 2-3 莱万特 复盘

- 预测：2-1，主胜，GOAL_BAND=3，BTTS=yes，主胜/平/客胜概率=48.0/28.0/24.0，confidence=63。
- 实际：2-3，客胜，总进球5，BTTS=yes。
- 命中与失败：BTTS 命中；方向、精确比分、总进球区间失败。
- 直接偏差：模型把“塞尔塔轻主 + 1-1低赔干扰 + 莱万特保级战意”折中为 2-1，但没有把 1-2/2-3 这类客队反打尾部纳入足够高的候选权重。
- 概率校准问题：主胜只有48%，并非强主；客胜24%不是极低尾部。最终输出却给出单点2-1，说明 direction edge 和 score edge 被混同。
- 总球问题：GOAL_BAND=3 把比赛压在2-3球；实际5球说明高比分尾部保护不足，尤其是“BTTS=yes + 保级战意 + 防线不稳”组合下，4+不应被过早降级。
- 风险标注其实已有：推荐理由/标签提到“1-1低赔强干扰、主胜升水、缺少外部市场核验”。这些信号应触发 C/D 降级、客胜反打备选和4+尾部审计，而不是仍输出单点2-1作为中心结论。

## 6. 代码问题

- P0：`async_call_ai_json` 此前在 HTTP 200 但 JSON 提取为空时仍标记 `ok=True/status=ok`，会把 AI 输出解析失败包装成正常调用，影响后续 fallback 与审计。
- P1：`build_evidence_packet` 在 `predict.py` 中出现基础版与 v20.3 包装版两个定义；当前有 `_BASE_BUILD_EVIDENCE_PACKET_V2021` 保存旧函数后再覆盖，运行上可用，但可维护性差，容易被后续编辑误删保存步骤。
- P1：默认 `AI_MAX_PROMPT_CHARS_PER_CHUNK=140000`，按中文/英文混合估算可达到3-4万 tokens，注意力稀释和成本风险都很高。
- P1：推荐降级只改 tier，不强制影响最终单点比分展示；因此 C 级高风险样本仍可能在前端被读成明确预测。
- P2：`GOAL_BAND` 只分 `0-1/2/3/4+`，没有对 BTTS+战意+防线不稳时的 4+ 尾部做独立保护分。

## 11. 升级方案

1. 协议层：AI 解析失败必须 fail-closed，禁止作为正常预测继续流转。
2. 概率层：当最大方向概率 <55% 时，单点比分必须带主备路径；主胜48%这种只能输出“主胜倾向/客胜反打风险”。
3. 比分层：把 0-1/1-2/2-3 客队反打与 4+ 高比分尾部作为独立审计表，不再只依赖 2-1/1-1中心比分。
4. 成本层：默认 prompt budget 从 140000 chars 降到 60000 chars，超过预算必须拆 chunk 或压缩历史上下文。
5. 数据层：落盘 web sources、published_at、accessed_at，并禁止把 `no_web_tool_available` 包装成已联网。
